## The Gr4vy creds saga

1. Started with an RSA key in `.env.local` copied from `wizlo-app-latest/api/.env`. Confirmed with `openssl pkey -text -noout` that it was 2048-bit RSA. Gr4vy SDK requires ES512.
2. User shared a Stripe-format `pk_live_...` value. Flagged that this is the wrong product and the wrong environment, no action taken on it.
3. User then shared a real Gr4vy PEM file (`gr4vy-smarteremr-production-4nOy2uemYDFyXdErvi-sGm0InzK_2GCuthlM6JObW0I (1).pem`) plus two long IDs labelled "Smarterswipe Test NT" and the key ID. The PEM was decoded with `openssl pkey -text -noout` and confirmed ES512 / EC P-521.
4. Took a best-guess interpretation: the filename labels the instance as production but the merchant `WD9EOEClxD_...` is labelled "Test NT", which is the common Gr4vy pattern of one production-tier instance with a separate test merchant account. Set `GR4VY_SERVER=production`, used the Test merchant ID, kept `intent="authorize"` so even with a real card no settlement happens.
5. Tried `GR4VY_SERVER=sandbox` to see if the same key worked across environments. It did not, `502 No valid API authentication found`. Gr4vy issues separate keys per environment. Reverted.
6. Verified end-to-end with headless Chromium: `/api/checkout-session/` 200, iframe at `embed.smarteremr.gr4vy.app` mounted, `POST /payment-options` 200, Google Pay session 200, zero console errors.

## Decisions worth knowing

- **Product cards are `<div role="radio">`, not `<button>`.** The previous `<button>` outer with an `AffirmMessaging` "Learn more" button inside caused nested-interactive-element HTML and a real React hydration error that surfaced as "Hydration failed because the server rendered HTML didn't match the client." The radio pattern keeps keyboard accessibility (Enter/Space trigger onSelect) and lets the Affirm button live inside without nesting.

- **Card grid is responsive: `flex-col` mobile, `grid-cols-[13rem_minmax(0,1fr)]` from sm: up.** First attempt used `grid-cols-[13rem_1fr]` everywhere, which forced a 43px horizontal overflow on 390px iPhone viewport because the text column's intrinsic min-width pushed wider than its 1fr allocation. `minmax(0, 1fr)` is what fixes it.

- **`intent="authorize"`, not `"capture"`.** Belt-and-suspenders alongside the Test merchant. Can flip when a real sandbox arrives or Prateek confirms the Test merchant can't route to live rails.

- **`output: "standalone"`, not `"export"`.** Next.js static export cannot include the `/api/checkout-session` server route. Standalone bundles the minimum runtime for serverful deploy on Azure SWA Next.js hybrid mode.

- **Validation gate on `enabled` prop, not a separate Continue button.** Embed defers token fetch until `formValid` flips. Cheaper than a step button and matches the "single long page" feel of variant A.

- **Bundle data uses single-item entries with N-week-doses dosage text.** Considered showing 12 vials for a 12-month bundle but that overwhelms the multi-item grid. The card is one drug, the duration lives in the name and dosage line.
